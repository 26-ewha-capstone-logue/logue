# Logue 🎙️

## 자연어 질문을 분석 가능한 구조로 바꾸는 Question-first 분석 지원 서비스

Logue는 데이터 전문 인력이 아닌 실무자의 모호한 질문을  
분석 조건으로 구조화하고, CSV 데이터 기반 결과와 계산 기준을 함께 제공하는 서비스입니다.

---

# 1. 문제 — 고객, Pain Point / Vitamin

### 발표 시간: 20초

## 고객

Logue의 핵심 고객은 데이터 분석가가 아니라,  
성과 수치를 보고 원인을 설명해야 하는 마케팅·기획·운영 실무자입니다.

## Pain Point

실무자는 “이번 주 전환율이 왜 떨어졌지?”처럼 업무 질문은 갖고 있지만,  
이를 BI 필터나 SQL 조건처럼 분석 가능한 형태로 바꾸는 데 어려움을 겪습니다.

| 상황 | 실제 문제 |
| --- | --- |
| 대시보드 확인 | 원하는 기준으로 바로 쪼개 보기 어려움 |
| 엑셀 재가공 | 기준일·지표 정의·필터가 흔들림 |
| 데이터팀 요청 | 질문을 다시 정리해 전달해야 함 |
| 결과 공유 | 숫자가 어떤 기준으로 계산됐는지 설명하기 어려움 |

## Painkiller vs Vitamin

| 구분 | 내용 |
| --- | --- |
| Vitamin | 예쁜 차트, 빠른 요약, 자동 리포트 |
| Painkiller | 모호한 질문을 분석 가능한 조건으로 바꾸고, 계산 기준까지 검증 가능하게 보여줌 |

> Logue가 해결하려는 문제는 “차트를 더 예쁘게 보여주는 것”이 아니라,  
> 실무자가 숫자를 설명하기 전 반드시 거쳐야 하는 질문 구조화와 기준 검증의 부담입니다.

# 2. 해결 아이디어

### 발표 시간: 20초

## 자연어 질문에서 출발해, 분석 조건과 결과를 함께 제공한다

Logue는 사용자가 자연어로 질문을 입력하면,  
질문 속에 숨어 있는 지표, 기준일, 비교 기간, 그룹 기준, 필터 조건을 먼저 구조화합니다.

예시 질문:

> “이번 주 가입 전환율이 지난주 대비 어디에서 가장 많이 떨어졌어?”

Logue가 해석하는 구조:

| 항목 | 예시 |
| --- | --- |
| 분석 유형 | ranking / comparison |
| 지표 | 가입 전환율 |
| 계산식 | signup_complete / landing_sessions |
| 기준일 | signup_date 또는 created_at |
| 비교 기간 | 이번 주 vs 지난주 |
| 그룹 기준 | channel, device, user_type |
| 필터 | internal_test 제외 |

## 핵심 아이디어

사용자가 직접 BI 조건을 설정하는 대신,  
Logue가 질문을 분석 가능한 구조로 번역하고  
그 기준을 사용자에게 보여준 뒤 결과를 제공합니다.

# 3. 기술 / 구현

### 발표 시간: 30초

## 사용자가 들으면 바로 떠올릴 수 있는 구현 흐름

~~~mermaid
flowchart LR
    A[사용자 질문 입력] --> B[AI가 질문 해석]
    B --> C[분석 조건 JSON 생성]
    C --> D[CSV 업로드]
    D --> E[컬럼 의미 매핑]
    E --> F[분석 실행]
    F --> G[표·차트·계산 기준 출력]
~~~

## 구현 방식

| 단계 | 구현 내용 |
| --- | --- |
| 1. 질문 입력 | 사용자가 자연어로 분석 질문 입력 |
| 2. 질문 해석 | AI가 질문을 `analysis_type`, `metric_id`, `group_by`, `filters` 등으로 변환 |
| 3. CSV 업로드 | 사용자가 분석할 데이터를 CSV로 업로드 |
| 4. 스키마 매핑 | CSV 컬럼명을 의미 기준으로 매핑 |
| 5. 분석 실행 | Python 기반으로 비교·순위 계산 수행 |
| 6. 결과 제공 | 결과값, 표, 차트, 계산 기준을 함께 출력 |

## 시스템 구조

~~~mermaid
flowchart TB
    U[User] --> FE[Frontend<br/>Next.js · TypeScript]
    FE --> BE[Backend<br/>Spring Boot]
    BE --> AI[AI Server<br/>FastAPI · OpenAI · pandas]
    AI --> BE
    BE --> FE
~~~

| 영역 | 역할 |
| --- | --- |
| Frontend | 질문 입력, CSV 업로드, 분석 기준 확인, 결과 화면 제공 |
| Backend | 요청 처리, 파일 관리, 분석 실행 흐름 제어 |
| AI Server | 자연어 질문 해석, 분석 조건 구조화, 모호성 판단 |

## 기술 핵심

| 기술 요소 | 설명 |
| --- | --- |
| Natural Language Parsing | 질문을 분석 조건 단위로 구조화 |
| Schema-aware Mapping | CSV 컬럼명을 의미 기준으로 해석 |
| Metric Grounding | 비즈니스 지표를 계산식과 연결 |
| Ambiguity Handling | 기준일·지표 충돌 시 경고 |
| Explainable Output | 결과와 계산 기준을 함께 제공 |

# 4. MVP / 배포

### 발표 시간: 20초

## MVP에서 만들어지는 것

초기 MVP는 모든 자유 질의를 처리하는 범용 분석기가 아니라,  
실무 반복성이 높은 두 가지 질문 유형에 집중합니다.

| MVP 범위 | 내용 |
| --- | --- |
| Comparison | 이번 주 vs 지난주처럼 기간 간 지표 비교 |
| Ranking | 어느 채널, 어느 디바이스에서 가장 많이 변했는지 순위화 |
| CSV 분석 | 별도 DB 연결 없이 업로드한 CSV 기반 분석 |
| 기준 노출 | 지표 정의, 기준일, 비교 기간, 필터 조건 표시 |
| 모호성 경고 | 기준이 애매할 경우 임의 계산 대신 확인 요청 |

## 최종 산출물

~~~mermaid
flowchart LR
    A[질문 입력 화면] --> B[CSV 업로드 화면]
    B --> C[분석 기준 확인 화면]
    C --> D[결과 표/차트 화면]
    D --> E[계산 기준 설명 화면]
~~~

## 배포 방법

| 영역 | 배포 방식 |
| --- | --- |
| Frontend | Vercel 또는 AWS 기반 웹 배포 |
| Backend | AWS EC2 / Docker 기반 배포 |
| AI Server | FastAPI 서버로 분리 배포 |
| Storage | CSV 파일 임시 저장 및 분석 후 결과 반환 |

> 사용자는 웹에서 질문을 입력하고 CSV를 업로드하면,  
> 분석 결과와 계산 기준을 한 화면에서 확인할 수 있습니다.

# 5. 차별성

### 발표 시간: 10초

## Logue는 BI도, 단순 AI 챗봇도 아닙니다

| 비교 대상 | 기존 방식 | Logue의 차이 |
| --- | --- | --- |
| BI 도구 | 사용자가 직접 조건을 설정해야 함 | 질문을 분석 조건으로 자동 구조화 |
| Excel | 수작업 재가공이 필요함 | 반복 질문을 구조화된 분석 흐름으로 처리 |
| AI 챗봇 | 그럴듯한 답을 생성할 수 있음 | 계산 기준과 지표 정의를 명시적으로 노출 |
| NLQ 분석 서비스 | 빠른 질의응답 중심 | 모호성 감지와 기준 검증에 집중 |

## 핵심 차별점

Logue의 차별성은 “자연어로 답해준다”가 아닙니다.  
**자연어 질문을 신뢰 가능한 분석 구조로 바꾸고, 그 계산 기준을 함께 보여준다는 점**입니다.

# 6. 산학트랙 PMF / 고객 인터뷰

### MD 파일에는 포함하되, 발표에서는 제외

## PMF 검증 방향

산학트랙에서는 완성된 제품을 바로 검증하기보다,  
먼저 핵심 고객이 실제로 이 문제를 반복적으로 겪는지 확인합니다.

## 고객 인터뷰 대상

| 대상 | 인터뷰 목적 |
| --- | --- |
| 마케팅 실무자 1명 | 전환율, 채널 성과, 캠페인 성과 분석 과정 확인 |
| 기획/운영 실무자 1명 | 지표 변화 원인 파악과 보고 과정 확인 |
| 데이터 협업 경험자 1명 | 데이터팀 요청 전후의 커뮤니케이션 병목 확인 |

## 인터뷰에서 확인할 질문

| 검증 항목 | 질문 |
| --- | --- |
| 문제 빈도 | 최근 2주 안에 수치를 직접 분석하거나 설명한 적이 있는가 |
| 현재 해결 방식 | 대시보드, 엑셀, 데이터팀 요청 중 무엇을 사용하는가 |
| 병목 지점 | 가장 오래 걸리는 단계가 데이터 확인인지, 기준 정리인지, 결과 설명인지 |
| 신뢰 문제 | 결과를 공유하기 전에 계산 기준을 다시 확인하는가 |
| 대체 가능성 | Logue가 질문 구조화와 기준 노출을 해준다면 기존 방식보다 나은가 |

## 인터뷰 결과 정리 양식

| 인터뷰 대상 | 확인된 Pain Point | 현재 대안 | Logue 적합성 |
| --- | --- | --- | --- |
| 마케팅 실무자 | 예: 채널별 전환율 하락 원인을 엑셀로 재가공 | BI + Excel | 높음 |
| 기획/운영 실무자 | 예: 기준일과 필터 조건을 매번 다시 확인 | 대시보드 + 수작업 | 중간~높음 |
| 데이터 협업 경험자 | 예: 질문을 데이터팀에 전달하기 전 정리가 어려움 | 슬랙/문서 요청 | 높음 |

## PMF 판단 기준

| 기준 | 판단 방식 |
| --- | --- |
| 문제 반복성 | 동일한 분석/보고 문제가 주기적으로 발생하는가 |
| 기존 방식의 불편함 | Excel, BI, 데이터팀 요청으로 해결하는 데 시간이 많이 드는가 |
| 지불/사용 의향 | 단순 호기심이 아니라 실제 업무 시간을 줄일 수 있다고 느끼는가 |
| MVP 적합성 | comparison / ranking 질문만으로도 초기 사용 가치가 있는가 |

> 단, 실제 인터뷰를 진행하지 않았다면 결과처럼 쓰면 안 됩니다.  
> 위 표는 인터뷰 설계 및 기록 양식으로 두고, 발표 전 실제 응답으로 채워야 합니다.

Logue는 데이터 전문 인력이 아닌 실무자가 던지는 모호한 질문을  
분석 가능한 구조로 바꾸고, CSV 기반 결과와 계산 기준을 함께 제공하는  
Question-first 분석 지원 서비스입니다.
