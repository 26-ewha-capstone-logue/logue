# Logue 🎙️ <img src="assets/logo.png" width="110" align="left" />
Logue는 데이터 전문 인력이 아닌 실무자의 모호한 질문을 자연어 기반 분석 구조로 변환해, CSV 데이터에서 신뢰할 수 있는 결과와 계산 기준을 함께 제공하는 분석 지원 서비스입니다.

## 🧭 프로젝트 개요

오늘날 실무자는 데이터를 직접 활용해야 하는 상황에 점점 더 자주 놓이고 있습니다.  
하지만 실제 업무 현장에서는 데이터가 여러 시스템에 흩어져 있고, 질문을 분석 가능한 형태로 바꾸는 과정 자체가 큰 부담이 됩니다.

Logue는 이 문제를 해결하기 위해, 사용자의 자연어 질문을 분석 구조로 변환하고,  
그 결과를 단순 수치가 아니라 **기준이 보이는 형태**로 제공하는 분석 경험을 제안합니다.

<br/>

## 🚨 문제 인식

### 🌍 1. 산업(상황 분석)

데이터 기반 의사결정은 이제 특정 분석 조직만의 일이 아니라, 마케팅·운영·기획 등 다양한 실무 영역으로 확산되고 있습니다.  
그러나 데이터는 CRM, 광고 플랫폼, 웹 로그, 내부 운영 도구 등 여러 시스템에 분산되어 축적되며, 실무자는 필요한 수치를 바로 보기보다 먼저 **어디에 어떤 데이터가 있는지**, **무엇을 기준으로 봐야 하는지**부터 판단해야 하는 상황에 놓입니다.

기존 BI 도구는 분산된 데이터를 통합하고 시각화하는 데 강점을 가지지만, 여전히 사용자가 **질문을 직접 분석 가능한 형태로 정리해야 한다는 전제**를 갖고 있습니다.  
그 결과 대시보드 확인 이후에도 엑셀 재가공, 데이터팀 요청, 기준 확인 같은 추가 작업이 반복되며, 빠른 의사결정에 필요한 마지막 단계가 여전히 비효율적으로 남아 있습니다.

### 🏢 2. 기업 분석

기존 BI 서비스는 정형 리포트와 모니터링에는 강하지만, 사용자가 던지는 모호한 질문을 즉시 실행 가능한 분석 조건으로 바꾸는 데에는 한계가 있습니다.  
예를 들어 “이번 주 전환율이 지난주 대비 어디에서 가장 많이 떨어졌어?” 같은 질문은 단순 조회가 아니라 **지표 정의, 날짜 기준, 비교 기간, 그룹 기준, 정렬 기준**이 함께 해석되어야 합니다.

또한 최근 등장한 자연어 기반 분석 서비스들도 빠른 응답에는 강점이 있지만, 실무 환경에서 중요한 **지표 정의의 일관성**, **계산 기준의 검증 가능성**, **모호성 처리** 측면에서는 아직 불안정한 경우가 많습니다.  
Logue는 이 지점을 문제로 보고, 단순히 답을 생성하는 것이 아니라 **조직이 신뢰할 수 있는 기준으로 질문을 해석하고 결과를 설명하는 경험**에 집중합니다.

### 👤 3. 소비자 분석

Logue의 핵심 사용자는 데이터 전문 인력이 아니라, 숫자를 보고 설명해야 하는 **실무자**입니다.  
이들은 분석 자체가 직무의 전부는 아니지만, 업무 과정에서 성과를 확인하고 원인을 설명하며 다음 액션을 정해야 합니다.

문제는 이들이 대체로 “무엇이 궁금한지”는 알고 있어도, 그것을 SQL이나 BI 필터 조건처럼 **분석 시스템이 이해할 수 있는 구조로 바꾸는 데 어려움**을 겪는다는 점입니다.  
결국 실무자는 질문을 더 잘게 쪼개고, 기준일과 지표 정의를 다시 맞추고, 결과가 맞는지 스스로 검증하는 과정을 반복하게 되며, 이 과정에서 시간과 신뢰가 함께 소모됩니다.

<br/>

## 💡 솔루션 (비즈니스 · 기획)

Logue는 사용자의 질문을 출발점으로 삼아, 해당 질문에 필요한 데이터 조건을 먼저 구조화한 뒤 분석 결과를 제공하는 **Question-first 분석 경험**을 제안합니다.  
핵심은 단순히 “답을 주는 것”이 아니라, **어떤 기준으로 그 답이 나왔는지 함께 보여주어 실무자가 결과를 신뢰하고 활용할 수 있게 만드는 것**입니다.

### 🎯 핵심 가치

| 구분 | 내용 |
| --- | --- |
| 질문 중심 분석 | 대시보드 탐색이 아니라 사용자의 자연어 질문에서 분석을 시작 |
| 기준 명확화 | 지표, 날짜 기준, 비교 기간, 그룹 기준, 필터를 구조화해 해석 |
| 모호성 제어 | 해석이 애매한 경우 임의 추론 대신 경고 및 수정 유도 |
| 결과 설명 가능성 | 숫자만 보여주는 것이 아니라 계산 기준과 해석 구조를 함께 제공 |
| 실무 적합성 | 데이터 전문가가 아닌 현업 실무자도 바로 사용할 수 있는 흐름 설계 |

### 🧩 주요 사용자 흐름

1. 사용자가 자연어로 질문을 입력한다.  
2. 시스템이 질문에서 필요한 지표·기준일·비교 기간·그룹 기준을 해석한다.  
3. 필요한 데이터 조건을 사용자에게 보여주고, 파일 업로드 또는 확인을 유도한다.  
4. 업로드된 CSV를 기준으로 분석을 수행한다.  
5. 결과를 표·차트·설명과 함께 제공하고, 필요 시 기준 수정이나 재질문을 지원한다.  

### 🛠 Logue가 해결하려는 핵심 문제

| 기존 방식의 한계 | Logue의 접근 |
| --- | --- |
| 질문을 사용자가 직접 BI 조건으로 번역해야 함 | 자연어 질문을 분석 구조로 변환 |
| 기준일·지표 정의가 흔들리면 결과 신뢰가 낮아짐 | 기준을 명시적으로 보여주고 검증 가능하게 설계 |
| 모호한 질문도 일단 답해버려 잘못된 결과를 줄 수 있음 | ambiguity를 감지하고 수정 흐름 제공 |
| 결과 숫자만 보여주고 왜 그렇게 계산됐는지 알기 어려움 | 계산 기준과 해석 구조를 함께 설명 |

## 🤖 솔루션 (기술)

Logue는 **FE / BE / AI**가 역할을 분리한 구조로 설계되어 있습니다.  
프론트엔드는 질문 입력과 결과 확인 경험을 담당하고, 백엔드는 요청 관리와 데이터 처리 흐름을 제어하며, AI는 질문 해석과 분석 기준 구조화를 담당합니다.

| 영역 | 역할 | 핵심 내용 |
| --- | --- | --- |
| FE | 사용자 인터페이스 | 질문 입력, 데이터 업로드, 분석 기준 확인, 결과 표/차트/설명 제공 |
| BE | 서비스 로직 및 API | 요청 처리, 파일 관리, 분석 실행 흐름 제어, 결과 전달 |
| AI | 질문 해석 및 구조화 | 자연어 질문을 `analysis_type`, `metric_id`, `date_field`, `group_by`, `filters` 등의 분석 조건으로 변환 |

### ⚙️ 기술적 차별점

| 항목 | 설명 |
| --- | --- |
| Natural Language Parsing | 사용자의 질문을 단순 키워드 매칭이 아니라 분석 조건 단위로 구조화 |
| Schema-aware Mapping | CSV마다 다른 컬럼명을 의미 기준으로 매핑 |
| Metric Grounding | “가입 전환율”처럼 비즈니스 지표를 컬럼 조합 기준으로 안정적으로 해석 |
| Ambiguity Handling | `signup_date` / `created_at`처럼 기준이 충돌할 때 경고 및 수정 유도 |
| Explainable Output | 결과값뿐 아니라 어떤 기준과 계산 방식으로 도출됐는지 함께 제시 |

### 🧪 현재 구현 방향

- **FE**: React 기반 인터페이스에서 질문, 테스트 케이스, 결과 시연 화면 제공
- **BE**: 데이터 처리 및 API 흐름 관리
- **AI**: OpenAI 기반 질문 해석 로직, 테스트 케이스 기준 구조화 및 결과 비교 실험 진행

## ✨ 핵심 기능

| 기능 | 설명 |
| --- | --- |
| 자연어 질문 입력 | 사용자가 분석 목적을 자연어로 입력 |
| 질문 해석 | 질문을 분석 가능한 조건(`analysis_type`, `metric_id`, `group_by` 등)으로 구조화 |
| CSV 업로드 분석 | 별도 DB 연결 없이 업로드한 CSV를 기준으로 분석 수행 |
| 스키마 의미 매핑 | 컬럼명이 달라도 의미 기준으로 역할을 추론하고 매핑 |
| 모호성 감지 | 날짜 기준, 지표 기준 등 해석 충돌 시 경고 및 수정 유도 |
| 결과 시각화 | 분석 결과를 표·차트·설명 형태로 제공 |
| 기준 노출 | 결과와 함께 어떤 계산 기준과 해석 구조를 사용했는지 제시 |
| 테스트 케이스 시연 | 동일 질문, 다른 표현 / 다른 스키마 / 모호한 상황에 대한 일관성 검증 |

## 🧪 테스트 범위

Logue의 초기 MVP는 모든 자유 질의를 처리하는 범용 분석기가 아니라,  
**비교(comparison)** 와 **순위(ranking)** 중심의 질문을 안정적으로 해석하는 데 집중합니다.

### ✅ 검증 대상

| 검증 대상 | 포함 여부 | 이유 |
| --- | --- | --- |
| `analysis_type` 도출 | ⭕ | comparison / ranking 구분은 해석의 출발점이기 때문 |
| `metric_id` 도출 | ⭕ | 중심 지표를 잘못 잡으면 결과 전체가 무너짐 |
| `date_field` 도출 | ⭕ | 기준일 충돌은 실무 해석에서 핵심 리스크 |
| `period_standard`, `period_compare` 도출 | ⭕ | 비교 질문의 핵심 조건 |
| `group_by` 도출 | ⭕ | “어디에서”라는 질문 의도를 해석하기 위해 필요 |
| `sort_by`, `sort_direction` 도출 | ⭕ | ranking / comparison 차이를 드러내는 핵심 필드 |
| `limit` 도출 | ⭕ | top n 질문 해석에 필수 |
| `filters` 도출 | ⭕ | `internal_test 제외` 같은 조건 반영 필요 |
| 차트/표 렌더링 | ❌ | 현재 단계의 핵심 검증 대상은 아님 |
| 인사이트 문장 생성 | ❌ | MVP 범위를 넘어서기 때문 |
| 자유형 모든 질문 | ❌ | 초기 단계에서는 범위를 좁혀 정확도 우선 |

### 🧩 테스트 관점

| 테스트 layer | 설명 |
| --- | --- |
| 정상 케이스 | 명확한 질문을 올바르게 구조화하는지 검증 |
| 표현 다양화 | 같은 의도를 다른 표현으로 넣어도 동일 기준이 나오는지 검증 |
| 스키마 변화 | 컬럼명이 달라도 같은 의미로 해석 가능한지 검증 |
| 실패/모호성 처리 | 애매한 상황에서 억지 추론 대신 경고·수정 유도를 하는지 검증 |

### 📈 기대 효과

이 테스트를 통해 Logue는 단순히 “질문에 답하는 서비스”가 아니라,  
**실무자가 실제로 신뢰하고 사용할 수 있는 분석 보조 도구**로 발전할 수 있는지 확인합니다.

## 🌱 확장 방향성

Logue는 단순 CSV 분석 데모에 머무르지 않고, 이후 더 넓은 분석 환경으로 확장될 수 있도록 설계하고 있습니다.

### 👥 1. 사용자 확장

초기에는 마케팅·기획·운영 직무처럼 숫자를 직접 확인하고 설명해야 하는 실무자를 주요 타겟으로 삼지만, 이후에는 재무, 영업, CS, HR 등 **비데이터 직군 전반의 셀프 분석 수요**로 확장할 수 있습니다.

### 🔌 2. 데이터 확장

현재는 CSV 업로드 기반으로 시작하지만, 향후에는 다양한 SaaS 및 데이터 소스와 연결해 **질문 → 데이터 선택 → 분석 실행**이 더 자연스럽게 이어지는 구조로 확장할 수 있습니다.

### 📊 3. 분석 기능 확장

초기 MVP는 비교·순위형 질문 중심으로 설계되었지만, 이후에는 추세 분석, 분포 분석, 세그먼트 비교, 지표 조합 분석 등으로 확장하여 더 폭넓은 실무 질문을 다룰 수 있습니다.

### 🛡 4. 신뢰성 확장

장기적으로는 조직별 지표 사전, metric registry, 기준 정의 관리 체계를 도입해 질문 해석의 유연성과 결과의 신뢰성을 동시에 확보하는 **조직형 분석 인프라**로 발전시키는 것을 목표로 합니다.
